# DOMAIN.md

## 1. Назначение проекта

Procurement API — это REST API сервиса закупок/маркетплейса, где:

* покупатели смотрят каталог, добавляют предложения магазинов в корзину и оформляют заказы;
* магазины управляют своими предложениями, ценами, остатками и обрабатывают позиции заказов;
* администраторы управляют пользователями, магазинами, каталогом, заказами и справочниками.

API должен поддерживать три основные роли:

* `buyer` — покупатель;
* `shop` — магазин / поставщик;
* `admin` — администратор платформы.

## 2. Общие правила пользователей

Пользователь хранится в модели `User`.

Бизнес-логин пользователя — `email`, используется для входа вместе с паролем.

Правила:

* `email` уникален.
* `username` не используется как логин.
* Пароль никогда не возвращается в API.
* Новый пользователь создаётся неактивным, если так как присутствует email-активация.
* JWT может получить только активный пользователь.
* Все защищённые endpoints требуют JWT.
* Чужие объекты возвращаются как `404 Not Found`, чтобы не раскрывать их существование.

## 3. Роли

## 3.1. Покупатель — `buyer`

Покупатель может:

* зарегистрироваться;
* активировать аккаунт;
* получить JWT;
* смотреть список товаров;
* фильтровать каталог;
* смотреть карточку предложения магазина;
* добавлять предложения магазинов в корзину;
* удалять позиции из своей корзины;
* создавать, смотреть и удалять свои адреса доставки;
* подтверждать свою корзину как заказ;
* смотреть историю своих заказов;
* смотреть детали своих заказов;
* отменять свой заказ до начала обработки магазином.

Покупатель не может:

* создавать товары;
* изменять цены;
* изменять остатки;
* управлять магазинами;
* видеть чужие заказы;
* видеть чужие адреса;
* изменять статусы заказов вручную, кроме разрешённой отмены;
* видеть административные данные.

## 3.2. Магазин / поставщик — `shop`

Пользователь с ролью `shop` представляет поставщика.

У пользователя-магазина должен быть связанный объект `Shop`.

Магазин может:

* смотреть профиль своего магазина;
* редактировать данные своего магазина;
* смотреть свои предложения;
* создавать предложения для существующих товаров;
* изменять цены и остатки своих предложений;
* архивировать свои предложения;
* смотреть позиции заказов, относящиеся только к его предложениям;
* принимать позиции заказа в работу;
* менять статус своих позиций заказа;
* отменять свою позицию заказа, если она ещё не отправлена.

Магазин не может:

* видеть предложения других магазинов в административном режиме;
* изменять чужие предложения;
* видеть полные заказы, если в них есть товары других магазинов;
* видеть персональные данные покупателя сверх необходимых для доставки;
* менять глобальный статус заказа напрямую;
* создавать глобальные категории без одобрения администратора;
* удалять товары из общего каталога;
* активировать себя без администратора.

## 3.3. Администратор — `admin`

Администратор — пользователь с `is_staff=True` и отдельным типом `admin`.

Администратор может:

* смотреть всех пользователей;
* блокировать и разблокировать пользователей;
* менять роль пользователя;
* смотреть все магазины;
* одобрять магазины;
* блокировать магазины;
* создавать, изменять и архивировать категории;
* создавать, изменять и архивировать товары;
* смотреть все предложения магазинов;
* архивировать некорректные предложения;
* смотреть все заказы;
* менять административные статусы заказов;
* отменять заказ;
* смотреть системные справочники;
* запускать или проверять импорт прайсов;
* смотреть статистику по заказам и магазинам.

Администратор не должен:

* видеть пароли;
* видеть JWT;
* логировать персональные данные без необходимости;
* изменять заказ так, чтобы нарушалась складская логика.

## 4. Магазины

## 4.1. Модель `Shop`

Магазин содержит:

* `id`;
* `name`;
* `url`;
* `owner`;
* `status`;
* `created_at`;
* `updated_at`.

Рекомендуемые статусы магазина:

* `pending` — ожидает проверки администратора;
* `active` — активен и может продавать;
* `blocked` — заблокирован;
* `archived` — архивирован.

Правила:

* У одного shop-пользователя может быть один активный магазин.
* Магазин может создавать предложения только после статуса `active`.
* Покупатели видят предложения только активных магазинов.
* Заблокированный магазин не может создавать и изменять предложения.
* Предложения заблокированного магазина скрываются из публичного каталога.
* Администратор может видеть все магазины независимо от статуса.

## 4.2. Регистрация магазина

Магазин регистрируется через отдельный endpoint.

После регистрации:

* создаётся пользователь с ролью `shop`;
* создаётся объект `Shop` со статусом `pending`;
* магазин не может продавать до одобрения администратора;
* администратор получает уведомление или видит заявку в списке.

## 5. Каталог

## 5.1. Основные сущности

* `Category` — категория товара.
* `Product` — глобальный товар каталога.
* `Shop` — магазин.
* `ProductInfo` — предложение конкретного магазина.
* `Parameter` — название характеристики.
* `ProductParameter` — значение характеристики предложения.

Ключевое правило:

`Product` — это общий товар.

`ProductInfo` — это конкретное предложение магазина с ценой, остатком и характеристиками.

Цена, остаток, магазин и параметры всегда относятся к `ProductInfo`, а не к `Product`.

## 5.2. Категории

Категории создаёт и изменяет администратор.

Магазин не может создавать категории напрямую.

Если магазину нужна новая категория, он может создать заявку на категорию.

Правила:

* категория может быть активной или архивной;
* архивные категории не показываются покупателям;
* товар не должен быть привязан к архивной категории для публичной продажи.

## 5.3. Товары

Товар `Product` создаёт администратор.

Магазин может:

* создать предложение для существующего товара;
* отправить заявку на создание нового товара.

Правила:

* один и тот же товар может продаваться в разных магазинах;
* в публичном списке товаров один `Product` отображается один раз;
* предложения разных магазинов не смешиваются;
* карточка предложения показывает конкретный `ProductInfo`.

## 5.4. Предложения магазина

`ProductInfo` содержит:

* `shop`;
* `product`;
* `name`;
* `quantity`;
* `price`;
* `price_rrc`;
* `status`;
* параметры предложения.

Рекомендуемые статусы предложения:

* `active` — доступно покупателям;
* `hidden` — скрыто магазином;
* `archived` — архивировано;
* `blocked` — заблокировано администратором.

Правила:

* магазин может изменять только свои предложения;
* покупатели видят только активные предложения активных магазинов;
* предложение с `quantity=0` может отображаться как недоступное или скрываться в зависимости от решения API;
* цена должна быть больше 0;
* остаток не может быть отрицательным;
* магазин не может изменить `shop` у предложения;
* администратор может заблокировать предложение.

## 5.5. Фильтрация каталога

Поддерживаемые фильтры:

* `search`;
* `category_id`;
* `shop_id`;
* `price_min`;
* `price_max`;
* `parameter`;
* `page`;
* `page_size`.

Критическое правило:

Фильтры `shop_id`, `price_min`, `price_max`, `parameter` должны совпадать на одном и том же `ProductInfo`.

Нельзя возвращать товар, если:

* магазин совпал у одного предложения;
* цена совпала у другого;
* параметр совпал у третьего.

Если `parameter` передан без двоеточия, API возвращает `400 Bad Request`.

## 6. Корзина

Корзина хранится как `Order` со статусом `basket`.

Позиция корзины — `OrderItem`.

`OrderItem` всегда ссылается на конкретный `ProductInfo`.

Правила:

* корзина принадлежит покупателю;
* текущая корзина — самый ранний заказ пользователя со статусом `basket`;
* если корзины нет, `POST /api/basket/` создаёт её;
* `GET /api/basket/` при отсутствии корзины возвращает пустой массив;
* в одной корзине могут быть предложения разных магазинов;
* одинаковый `ProductInfo` в корзине объединяется;
* одинаковый `Product` от разных магазинов не объединяется;
* quantity не может быть меньше 1;
* итоговое quantity не может превышать доступный остаток;
* добавление в корзину не списывает остаток;
* остаток списывается или резервируется только при подтверждении заказа.

## 7. Заказы

## 7.1. Общая модель заказа

`Order` — заказ покупателя.

`OrderItem` — позиция заказа, связанная с конкретным предложением магазина.

Так как один заказ может содержать товары разных магазинов, магазин работает не со всем заказом, а со своими `OrderItem`.

## 7.2. Статусы заказа

Рекомендуемые статусы `Order.state`:

* `basket` — корзина;
* `confirmed` — покупатель подтвердил заказ;
* `processing` — хотя бы один магазин принял позицию в работу;
* `sent` — все неотменённые позиции отправлены;
* `delivered` — все неотменённые позиции доставлены;
* `partially_canceled` — часть позиций отменена;
* `canceled` — весь заказ отменён.

Правила:

* `basket` нельзя выставлять через PATCH;
* покупатель может отменить заказ только до начала обработки;
* магазин не меняет статус всего заказа напрямую;
* статус заказа пересчитывается на основе статусов его позиций;
* администратор может отменить заказ вручную с обязательным указанием причины.

## 7.3. Статусы позиции заказа

Рекомендуемые статусы `OrderItem.state`:

* `confirmed` — позиция подтверждена покупателем;
* `accepted` — магазин принял позицию;
* `assembled` — позиция собрана;
* `sent` — позиция отправлена;
* `delivered` — позиция доставлена;
* `canceled` — позиция отменена.

Правила:

* после подтверждения заказа все позиции получают статус `confirmed`;
* магазин может менять статус только своих позиций;
* магазин не может вернуть позицию назад по статусу без администратора;
* магазин не может отменить позицию после отправки;
* покупатель не может менять статус позиции напрямую;
* администратор может изменить статус позиции вручную при спорной ситуации.

## 7.4. Подтверждение заказа

При подтверждении корзины:

* проверяется, что заказ принадлежит покупателю;
* проверяется, что заказ в статусе `basket`;
* проверяется, что в заказе есть позиции;
* проверяется, что контакт доставки принадлежит покупателю;
* проверяется актуальный остаток по каждому `ProductInfo`;
* остаток резервируется или списывается;
* заказ получает статус `confirmed`;
* все позиции получают статус `confirmed`;
* к заказу привязывается контакт;
* отправляются email-уведомления покупателю, администраторам и магазинам.

Если остатка недостаточно, API возвращает `400 Bad Request`.

Если заказ или контакт чужой, API возвращает `404 Not Found`.

## 7.5. Резервирование остатков
* добавить поле `reserved_quantity`;
* хранить резерв отдельно от физического остатка;
* освобождать резерв при отмене.

## 8. Контакты доставки

Контакт доставки принадлежит покупателю.

Поля:

* `city`;
* `street`;
* `house`;
* `structure`;
* `building`;
* `apartment`;
* `phone`.

Правила:

* покупатель видит только свои контакты;
* покупатель может удалить только свой контакт;
* чужой контакт возвращает `404 Not Found`;
* контакт, привязанный к оформленному заказу, лучше не удалять физически, а скрывать или архивировать;
* адрес и телефон нельзя логировать.

## 9. Email-уведомления

События для уведомлений:

* регистрация покупателя;
* регистрация магазина;
* одобрение магазина;
* блокировка магазина;
* подтверждение заказа;
* новая позиция заказа для магазина;
* изменение статуса позиции магазином;
* отмена заказа;
* отмена позиции;
* доставка заказа.

Правила:

* email покупателя не логируется;
* email администратора не логируется;
* если получателей нет, отправка пропускается с warning-логом;
* ошибка отправки письма не должна ломать заказ, если заказ уже подтверждён;
* критичные ошибки отправки нужно логировать без персональных данных.

## 10. Административные справочники

Администратор управляет:

* пользователями;
* магазинами;
* категориями;
* товарами;
* параметрами;
* предложениями;
* заказами;
* заявками магазинов;
* заявками на новые товары;
* заявками на новые категории.

Удаление важных сущностей лучше делать через архивирование.

Физическое удаление допустимо только для тестовых или ошибочных данных, если нет связанных заказов.

Администирование происходит через админ панель django, поэтому её надо настроить.

## 11. Логирование

Разрешено логировать:

* ID объектов;
* количество обработанных объектов;
* названия команд;
* факт создания заказа;
* факт изменения статуса;
* факт импорта прайса;
* технические ошибки.

Нельзя логировать:

* JWT;
* пароль;
* cookies;
* тело запроса;
* email;
* телефон;
* адрес доставки;
* персональные данные покупателя.

## 12. Общие правила ошибок

* `400 Bad Request` — неверные входные данные или бизнес-ограничение.
* `401 Unauthorized` — нет JWT или токен неверный.
* `403 Forbidden` — пользователь авторизован, но его роль не имеет доступа к endpoint.
* `404 Not Found` — объект не найден или не принадлежит пользователю.
* `409 Conflict` — конфликт состояния, например попытка подтвердить заказ при изменившемся остатке.
* `500 Internal Server Error` — непредвиденная ошибка.

Чужие объекты возвращать как `404`.

Недостаток роли возвращать как `403`.

## 13. Главные бизнес-инварианты

* Покупатель работает только со своими заказами, корзиной и контактами.
* Магазин работает только со своими предложениями и позициями заказов.
* Администратор видит всё.
* `Product` — общий товар.
* `ProductInfo` — предложение магазина.
* В корзину и заказ добавляется `ProductInfo`, а не `Product`.
* Предложения разных магазинов не объединяются.
* Фильтры предложения должны совпадать на одном `ProductInfo`.
* Остаток не может стать отрицательным.
* Заказ нельзя подтвердить без позиций.
* Заказ нельзя подтвердить с чужим адресом.
* Магазин не может менять весь заказ, только свои позиции.
* Покупатель не может видеть внутренние данные магазина.
* Магазин не может видеть товары и позиции других магазинов.
* Администраторские действия должны быть покрыты тестами доступа.
