# 1. Назначение DOMAIN.md ----

Этот файл описывает бизнес-логику Procurement API для AI-агента Codex.
Используй его как источник доменных правил перед изменением моделей, сервисов,
view-классов, serializer'ов, тестов и OpenAPI-документации.

Цель проекта: REST API сервиса закупок, где покупатель регистрируется,
получает JWT, смотрит каталог предложений магазинов, добавляет предложения в
корзину, сохраняет адрес доставки и подтверждает заказ.

# 2. Роли и пользователи ----

## 2.1. User ----

Пользователь находится в модели `api.models.User`.

Ключевые правила:

- Логин пользователя - уникальный `email`.
- `username` не является уникальным бизнес-логином.
- Новый пользователь по умолчанию создается с `type="buyer"`.
- Новый пользователь создается неактивным, потому что включена Djoser-активация.
- JWT может получить только активный пользователь.

## 2.2. Типы пользователей ----

В `USER_TYPE_CHOICES` есть:

- `buyer` - покупатель.
- `shop` - магазин или поставщик.

На текущем этапе бизнес-эндпоинты реализуют сценарии покупателя. Поставщик пока
не управляет прайсом через API; загрузка прайса идет через management command
`load_shop_data`.

# 3. Каталог ----

## 3.1. Основные сущности ----

- `Shop` - магазин, который продает товары.
- `Category` - категория каталога. Связана с магазинами через ManyToMany
  `Category.shops`.
- `Product` - общий товар каталога без цены, остатка и магазина.
- `ProductInfo` - конкретное предложение магазина по товару. Содержит `shop`,
  `product`, `name`, `quantity`, `price`, `price_rrc`.
- `Parameter` - название характеристики.
- `ProductParameter` - значение характеристики конкретного `ProductInfo`.

Главное разделение домена: `Product` не является предложением магазина.
Цена, остаток, магазин и характеристики находятся в `ProductInfo`.

## 3.2. Товар в нескольких магазинах ----

Один и тот же `Product` может иметь несколько `ProductInfo` от разных магазинов.
Например, один смартфон может продаваться в "Связной" и "ТехноМаркет" по разным
ценам и с разными остатками.

Правила:

- В списке каталога `GET /api/products/` возвращаются `Product`, а не
  `ProductInfo`.
- В детальном endpoint `GET /api/products/{id}/` параметр `{id}` - это ID
  `ProductInfo`, а не ID `Product`.
- Если один товар есть в двух магазинах, в списке товаров он должен появляться
  один раз.
- У каждого магазина должно быть отдельное предложение `ProductInfo`.
- Нельзя смешивать данные разных предложений одного товара.

## 3.3. Фильтры каталога ----

Поддерживаемые query-параметры:

- `search` - поиск по названию `Product.name`, без учета регистра.
- `category_id` - фильтр по категории `Product.category`.
- `shop_id` - фильтр по магазину предложения.
- `price_min` - нижняя граница цены предложения.
- `price_max` - верхняя граница цены предложения.
- `parameter` - фильтр по характеристике в формате `имя:значение`.
- `page`, `page_size` - пагинация.

Критически важное правило: фильтры `shop_id`, `price_min`, `price_max` и
`parameter` должны совпадать на одном и том же `ProductInfo`. Нельзя возвращать
товар, если `shop_id` совпал на предложении одного магазина, а цена или
характеристика совпала на предложении другого магазина.

Если `parameter` передан без двоеточия, API возвращает `400 Bad Request`.

# 4. Корзина ----

## 4.1. Модель корзины ----

Корзина хранится как `Order` со статусом `basket`.

Позиция корзины - это `OrderItem`, который ссылается на конкретный
`ProductInfo`. Это важно: позиция корзины добавляет не общий товар `Product`, а
предложение конкретного магазина.

## 4.2. Текущая корзина ----

Текущая корзина пользователя - самый ранний `Order` пользователя в статусе
`basket`, отсортированный по `id`.

Причина такого правила: в старой или поврежденной базе у пользователя могут
остаться несколько открытых корзин. Выбор первой корзины делает поведение
предсказуемым.

Если открытой корзины нет, `POST /api/basket/` создает новую корзину.
`GET /api/basket/` при отсутствии корзины возвращает пустой массив.

## 4.3. Добавление товара ----

Endpoint: `POST /api/basket/`.

Входные поля:

- `product_info_id` - ID предложения магазина.
- `quantity` - количество, минимум 1.

Правила:

- Если такой `ProductInfo` уже есть в текущей корзине, `OrderItem.quantity`
  увеличивается.
- Если в корзине есть тот же `Product`, но другой `ProductInfo`, создается
  отдельная позиция. Предложения разных магазинов не объединяются.
- Итоговое количество по позиции не может превышать `ProductInfo.quantity`.
- Если запрос превышает остаток, API возвращает `400 Bad Request`.
- Добавление в корзину не списывает складской остаток. Сейчас остаток только
  проверяется.

## 4.4. Удаление позиции ----

Endpoint: `DELETE /api/basket/?item_id=<id>`.

Правила:

- Удаляется именно `OrderItem`, а не `ProductInfo`.
- `product_info_id` временно поддерживается как устаревший alias для `item_id`.
- Пользователь может удалить только свою позицию из заказа в статусе `basket`.
- Попытка удалить чужую позицию или позицию из оформленного заказа возвращает
  `404 Not Found`.

# 5. Контакты доставки ----

Контакт доставки хранится в модели `Contact`.

Поля адреса:

- `city`
- `street`
- `house`
- `structure`
- `building`
- `apartment`
- `phone`

Правила:

- Контакт всегда принадлежит пользователю.
- `GET /api/contact/` возвращает только контакты текущего пользователя.
- Если контактов нет, API возвращает `{"data": []}`.
- `POST /api/contact/` создает контакт для текущего пользователя.
- `DELETE /api/contact/?id=<id>` удаляет только контакт текущего пользователя.
- Попытка удалить чужой контакт возвращает `404 Not Found`.

Не логируй адрес, телефон и тело запроса.

# 6. Заказы ----

## 6.1. Статусы заказа ----

Статусы находятся в `STATE_CHOICES`:

- `basket` - корзина.
- `new` - новый.
- `confirmed` - подтвержден.
- `assembled` - собран.
- `sent` - отправлен.
- `delivered` - доставлен.
- `canceled` - отменен.

Текущая реализация не задает строгий граф переходов между бизнес-статусами.
`PATCH /api/orders/{id}/` разрешает выставить любой статус из списка
`new`, `confirmed`, `assembled`, `sent`, `delivered`, `canceled`.
Статус `basket` через PATCH запрещен.

Если потребуется строгий workflow, добавляй его осознанно и покрывай тестами.

## 6.2. Подтверждение заказа ----

Endpoint: `POST /api/order/confirm/`.

Входные поля:

- `order_id` - ID заказа в статусе `basket`.
- `contact_id` - ID контакта текущего пользователя.

Правила:

- Заказ должен принадлежать текущему пользователю.
- Заказ должен быть в статусе `basket`.
- Заказ должен содержать хотя бы одну позицию.
- Контакт должен принадлежать текущему пользователю.
- После подтверждения заказ получает статус `confirmed`.
- К заказу привязывается выбранный контакт доставки.
- После подтверждения отправляются email-уведомления клиенту и администраторам.

Если корзина пустая, API возвращает `400 Bad Request`.
Если заказ или контакт чужой, API возвращает `404 Not Found`.

## 6.3. История заказов ----

Endpoint: `GET /api/orders/`.

Правила:

- Возвращаются только заказы текущего пользователя.
- Заказы в статусе `basket` исключаются.
- Ответ постраничный.
- `total_sum` считается как сумма `OrderItem.quantity * ProductInfo.price`.

## 6.4. Детали заказа ----

Endpoint: `GET /api/orders/{id}/`.

Правила:

- Заказ должен принадлежать текущему пользователю.
- Ответ включает `contact`, `items`, `total_sum`.
- Чужой заказ возвращается как `404 Not Found`.

# 7. Email-уведомления ----

Уведомления отправляет `api.management.email_service.send_order_confirmation`.

Правила:

- Если у клиента есть email, отправляется письмо клиенту.
- Если `ADMIN_EMAILS` не пустой, отправляется письмо администраторам.
- Если клиента или администраторов нет, отправка соответствующего письма
  пропускается и пишется warning-лог.
- Email-адреса не нужно логировать.

# 8. Загрузка прайсов ----

Management command:

```bash
python manage.py load_shop_data data/shop1.yaml
```

Бизнес-логика находится в `api.services.shop_data`.

## 8.1. Структура YAML ----

Ожидаемые секции:

- `shop` - обязательное имя магазина.
- `url` - необязательный URL магазина.
- `categories` - список категорий с `id` и `name`.
- `goods` - список предложений с категорией, названием, ценами, остатком и
  параметрами.

Если поле `shop` отсутствует, команда падает с `CommandError`.

## 8.2. Идемпотентность ----

Повторный запуск загрузки не должен создавать дубли.

Правила поиска и обновления:

- `Shop` ищется по `name`; URL обновляется, если изменился.
- `Category` ищется по `name`; магазин добавляется в `Category.shops`.
- `Product` ищется по `name`; категория обновляется.
- `ProductInfo` ищется по `(product, shop, name)`; остаток и цены обновляются.
- `ProductParameter` для предложения заменяются целиком при каждой загрузке.

## 8.3. Неизвестная категория ----

Если товар ссылается на категорию, которой нет в секции `categories`, товар
пропускается. Команда пишет сообщение в stderr и warning-лог.

# 9. Правила доступа и ошибок ----

Общее правило безопасности: бизнес-эндпоинты работают только с объектами
текущего JWT-пользователя.

Коды ответов:

- `401 Unauthorized` - пользователь не передал JWT или токен некорректен.
- `400 Bad Request` - ошибка валидации входных данных или бизнес-ограничение.
- `404 Not Found` - объект не найден или не принадлежит текущему пользователю.

Чужие объекты намеренно возвращаются как `404`, чтобы не раскрывать их
существование.

# 10. Логирование бизнес-логики ----

Логи нужны для разработки и диагностики.

Правила:

- Используй `debug` для веток выполнения: старт операции, выбранные ID,
  количество объектов, ключи фильтров.
- Используй `info` для изменений состояния: создана корзина, добавлена позиция,
  заказ подтвержден, письмо отправлено, прайс загружен.
- Используй `warning` для бизнес-отказов: превышен остаток, пустая корзина,
  пропущен товар, нет получателей email.
- Не логируй JWT, пароли, cookies, тело запроса, телефон, адрес доставки и
  email-адреса.

# 11. Что проверять при изменениях ----

При изменении доменной логики добавляй или обновляй тесты.

Минимальные регрессионные сценарии:

- Товар есть в двух магазинах и фильтры применяются к одному `ProductInfo`.
- Добавление одного `Product` из разных магазинов создает разные позиции.
- Повторное добавление одного `ProductInfo` увеличивает quantity.
- Нельзя добавить quantity больше остатка.
- Нельзя подтвердить пустую корзину.
- Нельзя читать, менять или удалять чужие заказы, контакты и позиции.
- Повторная загрузка YAML не создает дубли.
- Пропущенный товар с неизвестной категорией не создает `Product`.
