# Спецификация API сервиса автоматизации закупок

## 1. Назначение документа ----

Этот документ описывает целевую бизнес-логику и архитектуру backend API для
сервиса автоматизации закупок.

Документ нужен как единый договор для разработки:

- какие роли есть в системе;
- какие доменные сущности используются;
- какие сценарии должен поддерживать API;
- какие инварианты нельзя нарушать в коде, миграциях, serializers, services и tests;
- какие части текущей реализации нужно сохранить;
- какие части текущей реализации нужно переработать.

`AGENTS_SPEC.md` является проектной спецификацией. Если поведение кода меняется,
этот документ должен быть обновлен в том же изменении.

## 2. Контекст текущего проекта ----

Проект уже содержит рабочую основу закупочного API на Django, Django REST
Framework, Djoser, SimpleJWT, drf-spectacular и django-filter.

Текущая структура важной реализации:

- `api/models.py` - доменные модели пользователей, магазинов, каталога,
  предложений, контактов, заказов и позиций заказов;
- `api/views.py` - тонкие API views, которые делегируют бизнес-логику в сервисы;
- `api/services/*.py` - сервисный слой для корзины, заказов, магазинов, каталога,
  контактов, импорта и административных операций;
- `api/serializers.py` - входные команды и форматы ответов;
- `api/permissions.py` - ролевые permissions;
- `api/filters.py` - фильтрация каталога;
- `api/tests/*.py` - тесты ключевой бизнес-логики;
- `api/management/commands/load_shop_data.py` и `api/services/shop_data.py` -
  текущая загрузка YAML-прайсов через management command.

Сильные решения текущего кода, которые нужно сохранить:

- разделение `Product` и `ProductInfo`;
- сервисный слой между views и моделями;
- роли `buyer`, `shop`, `admin`;
- статусы магазинов, предложений, заказов и позиций заказа;
- резервирование остатков через `reserved_quantity`;
- пересчет статуса заказа из статусов позиций;
- soft-delete контактов, которые уже использовались в оформленных заказах;
- проверка чужих объектов через `404 Not Found`;
- OpenAPI-документация через drf-spectacular.

Главные проблемы текущей реализации, которые нужно исправлять итеративно:

- каталог сейчас смешивает `Product` и `ProductInfo`;
- детальный endpoint товара фактически принимает `ProductInfo.id`, хотя URL
  выглядит как endpoint `Product`;
- ответы API не везде имеют единый формат;
- импорт прайса пока management-команда, а не API-сценарий поставщика;
- импорт не использует стабильные поля `external_id` и `model`;
- заказ не хранит ценовой снимок позиции;
- не хватает уникальных DB-ограничений для конкурентных сценариев;
- API корзины и контактов использует query-параметры для удаления ресурсов;
- статус магазина перегружает модерацию, блокировку и прием заказов.

## 3. Архитектурная сессия команды ----

Перед фиксацией целевой архитектуры роли команды пришли к следующим выводам.

### 3.1. Backend architect

`Product` должен оставаться общей карточкой каталога, а `ProductInfo` должен
стать явным коммерческим предложением магазина.

Покупатель покупает не абстрактный товар, а конкретное предложение:

- конкретный магазин;
- конкретная цена;
- конкретный доступный остаток;
- конкретные характеристики;
- конкретный статус доступности.

Поэтому API должен быть offer-oriented там, где речь идет о покупке, корзине,
резерве и заказе.

### 3.2. Domain developer

Закупочный домен строится вокруг цепочки:

1. Поставщик заводит магазин.
2. Администратор модерирует магазин.
3. Поставщик создает предложения или импортирует прайс.
4. Покупатель выбирает доступные предложения.
5. Корзина хранит выбранные `ProductInfo`.
6. Подтверждение заказа резервирует остатки.
7. Поставщик обрабатывает только свои позиции.
8. Отправка позиции списывает физический остаток.
9. Заказ пересчитывает общий статус по позициям.

Эта цепочка должна быть видна в моделях, сервисах, API и тестах.

### 3.3. QA engineer

Нельзя ограничиться happy path. Обязательны тесты на:

- права доступа;
- чужие объекты;
- повторный импорт;
- конкурентные ограничения корзины;
- резерв и списание остатков;
- изменение цены после подтверждения заказа;
- soft-delete контактов;
- архивирование товаров и предложений;
- переходы статусов заказа и позиций.

### 3.4. Security reviewer

API не должен раскрывать чужие данные.

Правила безопасности:

- пароли, JWT, cookies, секреты и полные тела запросов с персональными данными не
  логируются;
- покупатель видит только свои корзины, заказы и контакты;
- поставщик видит только свой магазин, свои предложения и свои позиции заказов;
- поставщик не видит товары других поставщиков внутри смешанного заказа;
- администратор имеет расширенные права, но не получает секретные данные;
- чужой объект для buyer/shop возвращается как `404 Not Found`;
- недостаточная роль возвращает `403 Forbidden`.

### 3.5. Critic

Критик отклонил архитектуру, в которой каталог остается только списком `Product`.

Причины:

- список товаров без цены и магазина не решает закупочный сценарий;
- детальная карточка по `ProductInfo.id` под URL `products/<id>` вводит в
  заблуждение;
- фильтры по цене, магазину и характеристикам относятся к предложению, а не к
  абстрактному товару;
- история заказа без ценового снимка ломается при изменении цены поставщиком;
- импорт по названию товара создает дубли при переименовании.

Итоговое решение команды:

- сохранить `Product` как общий каталог;
- сделать `ProductInfo` явным `Offer`;
- в целевой API-карте разделить товары и предложения;
- добавить стабильные ключи импорта;
- добавить ценовой снимок в позицию заказа;
- описать статусы как автоматы, а не как свободное поле.

## 4. Общая идея приложения ----

Приложение представляет собой REST API для автоматизации закупок в розничной
или корпоративной сети.

Сервис работает без обязательного frontend-приложения. Все взаимодействие с
системой происходит через API-запросы и Swagger/OpenAPI-документацию.

В системе есть:

- покупатели, которые оформляют закупки;
- поставщики, которые управляют магазинами и предложениями;
- администраторы, которые модерируют платформу и управляют справочниками;
- единый каталог товаров;
- предложения магазинов с ценами, остатками и характеристиками;
- корзина покупателя;
- заказы, которые могут содержать позиции от разных поставщиков;
- статусы исполнения по каждой позиции заказа.

Главное доменное правило:

`Product` - это общий товар каталога. `ProductInfo` - это предложение конкретного
магазина. В корзину и заказ всегда добавляется `ProductInfo`, а не `Product`.

## 5. Роли пользователей ----

## 5.1. Покупатель `buyer` ----

Покупатель оформляет закупки через API.

Покупатель может:

- зарегистрироваться через Djoser;
- активировать аккаунт по email, если включена активация;
- авторизоваться и получить JWT;
- смотреть доступный каталог;
- смотреть карточку товара и доступные предложения;
- фильтровать предложения по магазину, категории, цене и характеристикам;
- добавлять предложения в корзину;
- менять количество позиций в корзине;
- удалять позиции из корзины;
- создавать и удалять свои контакты доставки;
- подтверждать свою корзину как заказ;
- смотреть историю своих заказов;
- смотреть детали своих заказов;
- отменять заказ до начала обработки поставщиком.

Покупатель не может:

- создавать товары общего каталога;
- менять цены и остатки;
- управлять магазином;
- загружать прайсы;
- видеть чужие корзины, заказы и контакты;
- менять статусы позиций заказа;
- пользоваться административными endpoint'ами.

## 5.2. Поставщик `shop` ----

Поставщик управляет своим магазином и своими предложениями.

Поставщик может:

- зарегистрировать магазин через отдельный endpoint;
- авторизоваться после активации пользователя и одобрения магазина;
- смотреть и обновлять профиль своего магазина;
- создавать предложения по существующим товарам;
- менять свои цены, остатки и статусы предложений;
- импортировать свой YAML-прайс через API;
- смотреть только свои позиции в оформленных заказах;
- переводить свои позиции вперед по статусам исполнения;
- отменять свою позицию до отправки, если бизнес-правило это разрешает.

Поставщик не может:

- видеть чужие предложения как объект управления;
- менять чужие цены и остатки;
- видеть смешанный заказ целиком, если в нем есть позиции других поставщиков;
- видеть лишние персональные данные покупателя;
- менять глобальный статус заказа напрямую;
- создавать или менять глобальные справочники без администратора;
- обрабатывать заказы, если магазин не активен или не принимает заказы.

## 5.3. Администратор `admin` ----

Администратор управляет платформой и справочниками.

Администратор может:

- просматривать пользователей;
- активировать и блокировать пользователей;
- управлять ролями с сохранением ролевых инвариантов;
- модерировать магазины;
- управлять категориями, товарами, параметрами и предложениями;
- смотреть все оформленные заказы;
- менять статусы заказов и позиций в аварийных сценариях;
- указывать причину административной отмены;
- контролировать импорт прайсов;
- пользоваться Django Admin.

Администратор не должен:

- видеть пароли;
- видеть или логировать JWT;
- нарушать складские инварианты;
- менять заказ так, чтобы история заказа стала недостоверной;
- назначать роль `shop` пользователю без связанного магазина;
- снимать `is_staff` у пользователя с ролью `admin`.

## 6. Доменные модули ----

Целевая архитектура должна быть разделена по доменным зонам.

## 6.1. `users/auth` ----

Отвечает за:

- регистрацию покупателя;
- регистрацию пользователя магазина;
- активацию пользователя;
- JWT-аутентификацию;
- профиль пользователя;
- ролевые permissions.

Текущая основа:

- Djoser endpoints в `/api/auth/`;
- кастомный `User`;
- `UserCreateSerializer`;
- `ShopRegistrationView`;
- permissions в `api/permissions.py`.

## 6.2. `catalog` ----

Отвечает за:

- категории;
- общие товары;
- параметры характеристик;
- фильтрацию и поиск;
- публичное или авторизованное чтение каталога.

Текущая основа:

- `Category`;
- `Product`;
- `Parameter`;
- `ProductParameter`;
- `ProductFilter`.

Целевое решение:

- `Product` показывает общую карточку товара;
- доступные предложения возвращаются отдельным endpoint'ом или вложенным блоком;
- фильтры по цене, магазину и характеристикам применяются к одному и тому же
  предложению.

## 6.3. `shops/offers` ----

Отвечает за:

- магазин поставщика;
- модерацию магазина;
- прием заказов магазином;
- предложения магазина;
- импорт прайса.

Текущая основа:

- `Shop`;
- `ProductInfo`;
- `ShopProfileView`;
- `ShopOfferListCreateView`;
- `ShopOfferDetailView`;
- `shop_service`.

Целевое решение:

- `ProductInfo` в API следует называть `Offer`;
- добавить `external_id` и `model`;
- добавить стабильный ключ импорта `shop + external_id`;
- отделить статус модерации магазина от добровольной остановки приема заказов.

## 6.4. `basket` ----

Отвечает за текущую корзину покупателя.

Текущая основа:

- корзина хранится как `Order(state="basket")`;
- позиции корзины хранятся в `OrderItem(state="basket")`;
- `basket_service`.

Целевое решение:

- у покупателя должна быть одна активная корзина;
- `GET /basket/` должен возвращать объект корзины, позиции и суммы;
- операции над позициями должны быть ресурсными;
- одинаковый `ProductInfo` в корзине не дублируется.

## 6.5. `orders/fulfillment` ----

Отвечает за:

- подтверждение корзины;
- резервирование остатков;
- обработку позиций поставщиками;
- списание остатков при отправке;
- отмену заказов и позиций;
- пересчет общего статуса заказа.

Текущая основа:

- `Order`;
- `OrderItem`;
- `confirm_order`;
- `update_shop_order_item_state`;
- `recalculate_order_state`;
- `release_item_reserve`;
- `ship_item_stock`.

Целевое решение:

- подтверждение заказа резервирует остаток;
- физическое списание происходит при переводе позиции в `sent`;
- заказ хранит ценовые снимки позиций;
- общий статус заказа вычисляется по позициям;
- переходы статусов описаны как state machine.

## 6.6. `contacts` ----

Отвечает за адреса доставки покупателя.

Текущая основа:

- `Contact`;
- `contacts_service`;
- soft-delete для контактов, связанных с оформленными заказами.

Целевое решение:

- endpoint'ы должны быть ресурсными: `/contacts/`, `/contacts/{id}/`;
- использованный контакт нельзя физически удалять;
- контакт должен быть недоступен в новых заказах после soft-delete.

## 6.7. `admin_api` ----

Отвечает за backoffice API.

Текущая основа:

- `AdminUserListView`;
- `AdminShopListView`;
- `AdminCategoryListCreateView`;
- `AdminProductListCreateView`;
- `AdminOfferListView`;
- `AdminOrderListView`;
- `admin_service`.

Целевое решение:

- административные операции должны идти через сервисы;
- изменения статусов должны использовать те же инварианты, что и обычный API;
- аварийные изменения должны логироваться;
- административная отмена заказа требует причину.

## 6.8. `notifications/tasks` ----

Отвечает за:

- email-уведомления;
- будущие Celery-задачи;
- импорт и экспорт в фоне.

Целевое решение:

- email не должен откатывать подтвержденный заказ;
- отправка email должна запускаться через `transaction.on_commit`;
- тяжелые операции импорта и экспорта нужно вынести в Celery после стабилизации
  синхронного API.

## 7. Основные сущности данных ----

## 7.1. User ----

`User` - пользователь системы.

Ключевые поля:

- `id`;
- `email`;
- `password`;
- `first_name`;
- `last_name`;
- `company`;
- `position`;
- `type`;
- `is_active`;
- `is_staff`;
- `is_superuser`.

Бизнес-правила:

- email является логином и должен быть уникальным;
- пароль хранится только в виде hash;
- пароль никогда не возвращается в API;
- роль пользователя определяет доступные действия;
- неактивный пользователь не получает JWT;
- `type="admin"` требует `is_staff=True`;
- пользователь с ролью `shop` должен иметь связанный `Shop`;
- нельзя менять роль владельца магазина на не-shop без переназначения магазина.

## 7.2. Shop ----

`Shop` - магазин или поставщик.

Текущие поля:

- `id`;
- `owner`;
- `name`;
- `url`;
- `status`;
- `created_at`;
- `updated_at`.

Целевые поля:

- `is_accepting_orders` - принимает ли магазин новые заказы;
- `status` - административный статус модерации.

Статусы `status`:

- `pending` - магазин создан, но еще не одобрен;
- `active` - магазин одобрен;
- `blocked` - магазин заблокирован администратором;
- `archived` - магазин выведен из эксплуатации.

Бизнес-правила:

- магазин принадлежит одному пользователю-поставщику;
- один пользователь-поставщик управляет одним магазином;
- создавать и обновлять предложения может только владелец магазина;
- покупка доступна только если `status="active"` и `is_accepting_orders=True`;
- заблокированный или архивный магазин не появляется в покупательской выдаче;
- поставщик не может самостоятельно снять административную блокировку.

Если поле `is_accepting_orders` еще не реализовано, текущая реализация должна
считать прием заказов разрешенным только при `status="active"`.

## 7.3. Category ----

`Category` - категория товара.

Ключевые поля:

- `id`;
- `name`;
- `shops`;
- `status`.

Бизнес-правила:

- категория может быть связана с несколькими магазинами;
- вложенные категории в базовой реализации не обязательны;
- архивная категория скрывает товары и предложения из покупательского каталога;
- поставщик не управляет глобальными категориями напрямую;
- импорт может связывать существующую категорию с магазином;
- создание новой категории через импорт требует выбранной политики модерации.

## 7.4. Product ----

`Product` - общий товар каталога.

Ключевые поля:

- `id`;
- `category`;
- `name`;
- `status`.

Бизнес-правила:

- `Product` не хранит цену, остаток и магазин;
- `Product` может иметь много предложений от разных магазинов;
- покупатель не добавляет `Product` в корзину напрямую;
- архивный `Product` скрывает все свои предложения из покупательского каталога;
- изменение названия `Product` не должно ломать историю оформленных заказов.

## 7.5. ProductInfo / Offer ----

`ProductInfo` - текущее имя модели. В API и доменной речи сущность следует
называть `Offer`.

Ключевые текущие поля:

- `id`;
- `product`;
- `shop`;
- `name`;
- `quantity`;
- `reserved_quantity`;
- `price`;
- `price_rrc`;
- `status`.

Целевые поля:

- `external_id` - стабильный ID предложения в прайсе поставщика;
- `model` - модель или артикул из прайса;
- `updated_at` - время последнего обновления предложения.

Статусы:

- `active` - предложение доступно к покупке;
- `hidden` - скрыто поставщиком;
- `archived` - больше не используется;
- `blocked` - заблокировано администратором.

Бизнес-правила:

- цена и остаток относятся только к `ProductInfo`;
- доступный остаток равен `quantity - reserved_quantity`;
- `quantity >= 0`;
- `reserved_quantity >= 0`;
- `reserved_quantity <= quantity`;
- `price > 0`;
- `price_rrc >= 0`;
- в корзину добавляется только активное предложение активного магазина;
- поставщик не может уменьшить `quantity` ниже уже зарезервированного количества;
- импорт обновляет предложение по стабильному ключу `shop + external_id`;
- если `external_id` отсутствует в старых данных, временно допустим legacy-ключ,
  но его нужно считать техническим долгом.

## 7.6. Parameter ----

`Parameter` - название характеристики.

Примеры:

- цвет;
- диагональ;
- память;
- разрешение;
- объем накопителя.

Бизнес-правила:

- параметр описывает только имя характеристики;
- имя параметра должно быть уникальным после нормализации;
- параметр может использоваться у разных предложений;
- параметры создаются администратором или импортом по выбранной политике.

## 7.7. ProductParameter ----

`ProductParameter` - значение характеристики конкретного предложения.

Ключевые поля:

- `product_info`;
- `parameter`;
- `value`.

Бизнес-правила:

- характеристика относится к конкретному `ProductInfo`;
- один параметр не должен дублироваться у одного `ProductInfo`;
- при повторном импорте характеристики предложения заменяются актуальным набором;
- характеристики используются в карточке предложения и в фильтрах.

## 7.8. Contact ----

`Contact` - контакт доставки покупателя.

Ключевые поля:

- `id`;
- `user`;
- `city`;
- `street`;
- `house`;
- `structure`;
- `building`;
- `apartment`;
- `phone`;
- `is_deleted`.

Бизнес-правила:

- контакт принадлежит одному покупателю;
- покупатель видит только свои контакты с `is_deleted=False`;
- покупатель не может использовать чужой контакт;
- чужой контакт возвращает `404 Not Found`;
- контакт, привязанный к оформленному заказу, скрывается через soft-delete;
- контакт, не привязанный к оформленным заказам, можно удалить физически;
- адрес и телефон не логируются без необходимости.

## 7.9. Order ----

`Order` - корзина или оформленный заказ.

Ключевые поля:

- `id`;
- `user`;
- `dt`;
- `state`;
- `contact`;
- `cancellation_reason`.

Целевые поля:

- `confirmed_at`;
- `updated_at`;
- `total_amount_snapshot`, если сумма заказа фиксируется на уровне заказа.

Бизнес-правила:

- корзина хранится как `Order(state="basket")`;
- у покупателя должна быть одна активная корзина;
- оформленный заказ создается из корзины;
- заказ может содержать позиции разных магазинов;
- заказ нельзя подтвердить без позиций;
- заказ нельзя подтвердить без своего активного контакта;
- статус оформленного заказа должен пересчитываться из позиций;
- `basket` не должен попадать в историю заказов;
- отмена заказа освобождает резерв по еще не отправленным позициям.

## 7.10. OrderItem ----

`OrderItem` - позиция корзины или заказа.

Ключевые текущие поля:

- `id`;
- `order`;
- `product_info`;
- `quantity`;
- `state`.

Целевые поля для истории:

- `unit_price`;
- `price_rrc_snapshot`;
- `product_name_snapshot`;
- `offer_name_snapshot`;
- `shop_name_snapshot`;
- `external_id_snapshot`.

Бизнес-правила:

- позиция всегда связана с конкретным `ProductInfo`;
- одинаковый `ProductInfo` в одной корзине не дублируется;
- количество должно быть больше нуля;
- количество в корзине не может превышать доступный остаток;
- при подтверждении заказа фиксируется ценовой снимок;
- после подтверждения сумма заказа не меняется из-за изменения `ProductInfo.price`;
- при переводе позиции в `sent` физический остаток списывается;
- отмена позиции до отправки освобождает резерв;
- отправленную или доставленную позицию нельзя вернуть к статусам до отправки без
  отдельной возвратной операции, которой пока нет в базовой архитектуре.

## 8. Статусы и автоматы состояний ----

## 8.1. Статусы заказа ----

Допустимые статусы `Order.state`:

- `basket` - текущая корзина;
- `confirmed` - покупатель подтвердил заказ, остатки зарезервированы;
- `processing` - хотя бы одна позиция принята или собирается поставщиком;
- `sent` - все неотмененные позиции отправлены или доставлены, но не все доставлены;
- `delivered` - все неотмененные позиции доставлены;
- `partially_canceled` - часть позиций отменена, часть продолжает исполнение;
- `canceled` - все позиции отменены или заказ отменен полностью.

Правило:

после подтверждения общий статус заказа должен вычисляться из `OrderItem.state`,
а не редактироваться произвольно. Административное изменение допускается только
через сервис, который сохраняет складские инварианты.

## 8.2. Статусы позиции заказа ----

Допустимые статусы `OrderItem.state`:

- `basket` - позиция находится в корзине;
- `confirmed` - позиция подтверждена и зарезервирована;
- `accepted` - магазин принял позицию в работу;
- `assembled` - позиция собрана;
- `sent` - позиция отправлена, физический остаток списан;
- `delivered` - позиция доставлена;
- `canceled` - позиция отменена.

## 8.3. Переходы позиции поставщиком ----

Поставщик может двигать только свои позиции.

Разрешенные переходы:

| Текущий статус | Новый статус | Кто может выполнить | Складское действие |
| --- | --- | --- | --- |
| `confirmed` | `accepted` | shop/admin | нет |
| `accepted` | `assembled` | shop/admin | нет |
| `assembled` | `sent` | shop/admin | списать `quantity` из `reserved_quantity` и `quantity` |
| `sent` | `delivered` | shop/admin | нет |
| `confirmed` | `canceled` | shop/admin | освободить резерв |
| `accepted` | `canceled` | shop/admin | освободить резерв |
| `assembled` | `canceled` | shop/admin | освободить резерв |

Запрещено:

- менять `basket`-позицию через поставщицкий API;
- двигать позицию назад;
- отменять позицию после `sent`;
- переводить чужую позицию;
- менять позицию, если магазин не является владельцем `product_info`.

## 8.4. Отмена заказа покупателем ----

Покупатель может отменить заказ только если:

- заказ принадлежит ему;
- заказ в `confirmed` или `partially_canceled`;
- все неотмененные позиции еще в `confirmed`;
- поставщики еще не начали обработку.

При отмене:

- все неотмененные позиции становятся `canceled`;
- резерв освобождается;
- заказ становится `canceled`;
- покупателю отправляется уведомление, если уведомления включены.

## 8.5. Административные переходы ----

Администратор может выполнять аварийные переходы, но только через сервисы.

Правила:

- отмена заказа требует `cancellation_reason`;
- отмена до отправки освобождает резерв;
- перевод неотправленной позиции в `sent` или `delivered` списывает остаток;
- нельзя вернуть `sent` или `delivered` к статусам до отправки;
- все административные действия логируются по ID объектов.

## 9. Каталог и предложения ----

## 9.1. Доступ к каталогу ----

Целевое решение для проекта: каталог доступен авторизованным пользователям.

Причина:

- текущие настройки DRF требуют JWT по умолчанию;
- текущие тесты закрепляют `401 Unauthorized` для неавторизованного доступа;
- закупочный сервис может быть внутренним B2B-инструментом.

Если в будущем каталог нужно сделать публичным, это должно быть отдельным
изменением в permissions, тестах, OpenAPI и документации.

## 9.2. Целевая модель выдачи ----

Для покупателя должны быть доступны два уровня чтения:

1. Список товаров `Product` как общий каталог.
2. Список предложений `Offer/ProductInfo`, которые можно купить.

Рекомендуемая API-модель:

- `GET /api/products/` - список общих товаров с агрегированной информацией;
- `GET /api/products/{product_id}/` - карточка общего товара;
- `GET /api/products/{product_id}/offers/` - доступные предложения товара;
- `GET /api/offers/` - список доступных предложений с фильтрами;
- `GET /api/offers/{offer_id}/` - карточка конкретного предложения.

Допустимый промежуточный вариант:

- сохранить текущие URL на время перехода;
- явно описать в OpenAPI, что `products/<id>` возвращает `ProductInfo`;
- добавить новые offer-endpoint'ы;
- затем удалить или переименовать legacy endpoint.

## 9.3. Фильтрация ----

Фильтры:

- `search` - поиск по названию товара или предложения;
- `category_id`;
- `shop_id`;
- `price_min`;
- `price_max`;
- `parameter=name:value`;
- `in_stock`;
- `ordering`.

Главное правило фильтрации:

если фильтры относятся к предложению, они должны применяться к одному и тому же
`ProductInfo`.

Нельзя считать товар подходящим, если магазин совпал у одного предложения, цена
у другого, а характеристика у третьего.

## 9.4. Карточка предложения ----

Карточка предложения должна возвращать:

- `offer_id`;
- `product_id`;
- название товара;
- название предложения;
- магазин;
- модель или артикул;
- цену;
- рекомендованную цену;
- общий остаток;
- зарезервированный остаток, только для shop/admin;
- доступный остаток;
- статус предложения;
- характеристики;
- можно ли добавить в корзину.

## 10. Импорт прайсов ----

## 10.1. Назначение импорта ----

Импорт нужен для обновления предложений поставщика:

- цен;
- остатков;
- моделей;
- характеристик;
- статусов доступности.

Текущая management-команда должна остаться как служебный инструмент для
демонстрационных данных, но бизнес-сценарий поставщика должен идти через API.

## 10.2. Целевые endpoint'ы импорта ----

Рекомендуемые endpoint'ы:

- `POST /api/shop/imports/` - запустить импорт прайса;
- `GET /api/shop/imports/` - история импортов магазина;
- `GET /api/shop/imports/{id}/` - результат конкретного импорта.

Минимальный синхронный вариант:

- `POST /api/shop/imports/` принимает файл или URL;
- валидирует права и структуру;
- выполняет импорт в транзакции;
- возвращает отчет.

Продвинутый вариант:

- endpoint только создает `ImportJob`;
- Celery выполняет импорт в фоне;
- API возвращает статус задачи.

## 10.3. Формат YAML ----

Ожидаемая структура:

```yaml
shop: Example shop
url: https://example.test
categories:
  - id: 224
    name: Смартфоны
goods:
  - id: 4216292
    category: 224
    model: apple/iphone/xs-max
    name: Смартфон Apple iPhone XS Max 512GB
    price: 110000
    price_rrc: 116990
    quantity: 14
    parameters:
      Цвет: золотистый
      Встроенная память: 512
```

Правила:

- `goods[].id` должен сохраняться в `ProductInfo.external_id`;
- `goods[].model` должен сохраняться в `ProductInfo.model`;
- `shop` должен соответствовать магазину текущего поставщика или быть проверен
  политикой импорта;
- цена должна быть больше нуля;
- остаток не может быть отрицательным;
- неизвестная категория должна обрабатываться по выбранной политике.

## 10.4. Политика ошибок импорта ----

Целевое решение: fail-fast atomic import.

Это значит:

- импорт выполняется внутри транзакции;
- если файл невалиден, база не меняется;
- если товар ссылается на неизвестную категорию, импорт падает ошибкой;
- если цена или остаток невалидны, импорт падает ошибкой;
- если YAML не читается, импорт падает ошибкой;
- API возвращает понятную ошибку с количеством найденных проблем.

Причина:

для закупочного сервиса частично обновленный прайс опаснее, чем полный отказ
импорта.

Допустимый временный legacy-режим:

- management-команда может продолжать частично пропускать строки для загрузки
  демонстрационных данных;
- API-импорт поставщика должен быть атомарным.

## 10.5. Стратегия обновления ----

Ключ обновления предложения:

- `shop`;
- `external_id`.

Алгоритм:

1. Проверить права: только владелец активного магазина.
2. Прочитать YAML.
3. Валидировать обязательные поля.
4. Проверить категории.
5. Создать или обновить `Product`.
6. Создать или обновить `ProductInfo` по `shop + external_id`.
7. Обновить цену, остаток, модель, название и статус.
8. Заменить характеристики предложения.
9. Предложения магазина, отсутствующие в новом прайсе, перевести в `hidden` или
   `archived` по выбранной политике.
10. Вернуть отчет.

Рекомендуемая политика отсутствующих предложений:

- если предложение исчезло из прайса, переводить в `hidden`;
- архивировать только явным действием поставщика или администратора.

## 11. Корзина ----

## 11.1. Общая логика ----

Корзина - это `Order` со статусом `basket`.

Правила:

- корзина принадлежит одному покупателю;
- у покупателя должна быть одна активная корзина;
- корзина содержит `OrderItem`;
- каждая позиция ссылается на `ProductInfo`;
- добавление товара в корзину не резервирует складской остаток;
- остаток проверяется при добавлении и повторно при подтверждении.

## 11.2. Целевые endpoint'ы ----

Рекомендуемая карта:

- `GET /api/basket/` - получить текущую корзину;
- `POST /api/basket/items/` - добавить предложение;
- `PATCH /api/basket/items/{item_id}/` - изменить количество;
- `DELETE /api/basket/items/{item_id}/` - удалить позицию;
- `DELETE /api/basket/` - очистить корзину.

Текущий legacy endpoint `DELETE /api/basket/?item_id=...` допустим только на
переходный период.

## 11.3. Ответ корзины ----

`GET /api/basket/` должен возвращать:

- `order_id`;
- `state`;
- `items`;
- сумму каждой позиции;
- общую сумму;
- доступность каждой позиции;
- предупреждения, если предложение стало недоступно.

Пример логической структуры:

```json
{
  "id": 10,
  "state": "basket",
  "items": [
    {
      "id": 100,
      "offer_id": 55,
      "product_name": "Смартфон",
      "shop_name": "Main shop",
      "unit_price": 1000,
      "quantity": 2,
      "line_total": 2000,
      "available_quantity": 8
    }
  ],
  "total": 2000
}
```

## 11.4. Добавление позиции ----

Покупатель передает:

- `offer_id`;
- `quantity`.

Система проверяет:

- пользователь авторизован;
- пользователь имеет роль `buyer`;
- предложение существует;
- предложение активно;
- магазин активен и принимает заказы;
- товар и категория активны;
- количество больше нуля;
- итоговое количество в корзине не превышает доступный остаток.

Если позиция уже есть в корзине, количество увеличивается.

## 11.5. Изменение количества ----

При изменении количества:

- позиция должна принадлежать текущей корзине пользователя;
- новое количество должно быть больше нуля;
- новое количество не должно превышать доступный остаток;
- изменение не должно создавать вторую позицию с тем же предложением.

## 11.6. Удаление позиции ----

Удалить можно только позицию текущей корзины пользователя.

Чужая позиция или позиция оформленного заказа возвращает `404 Not Found`.

## 12. Контакты доставки ----

## 12.1. Endpoint'ы ----

Целевая карта:

- `GET /api/contacts/` - список контактов текущего покупателя;
- `POST /api/contacts/` - создать контакт;
- `GET /api/contacts/{id}/` - получить контакт;
- `PATCH /api/contacts/{id}/` - изменить контакт, если он не удален;
- `DELETE /api/contacts/{id}/` - удалить или скрыть контакт.

Текущий legacy endpoint `/api/contact/?id=...` допустим только на переходный
период.

## 12.2. Правила ----

- контакт принадлежит покупателю;
- список не показывает `is_deleted=True`;
- подтверждение заказа принимает только контакт текущего покупателя;
- использованный в оформленном заказе контакт скрывается, а не удаляется;
- неиспользованный контакт можно удалить физически;
- удаленный контакт нельзя использовать для нового заказа.

## 13. Подтверждение заказа ----

## 13.1. Условия подтверждения ----

Покупатель может подтвердить только свою корзину.

Для подтверждения нужны:

- `order_id`;
- `contact_id`.

Система проверяет:

1. Пользователь авторизован.
2. Пользователь имеет роль `buyer`.
3. Заказ принадлежит пользователю.
4. Заказ находится в `basket`.
5. В заказе есть позиции.
6. Контакт принадлежит пользователю и не удален.
7. Все предложения существуют.
8. Магазины активны и принимают заказы.
9. Товары и категории активны.
10. Остатков хватает для всех позиций.
11. Количество каждой позиции больше нуля.

## 13.2. Действия при подтверждении ----

Подтверждение выполняется атомарно.

Система должна:

1. Заблокировать строки позиций и предложений для конкурентного обновления.
2. Повторно проверить доступный остаток.
3. Увеличить `reserved_quantity` у каждого предложения.
4. Записать ценовой снимок в каждую позицию.
5. Перевести позиции в `confirmed`.
6. Привязать контакт к заказу.
7. Перевести заказ в `confirmed`.
8. Сохранить время подтверждения.
9. После commit отправить email-уведомления.
10. Вернуть детали оформленного заказа.

## 13.3. Ошибки подтверждения ----

API возвращает ошибку, если:

- корзина пуста;
- корзина уже подтверждена;
- контакт чужой или удален;
- предложение скрыто, архивировано или заблокировано;
- магазин неактивен или не принимает заказы;
- товар или категория архивированы;
- остатка не хватает;
- параллельный запрос уже зарезервировал остаток.

При ошибке:

- заказ остается в `basket`;
- резерв не меняется частично;
- ценовые снимки не записываются частично.

## 14. Заказы и исполнение ----

## 14.1. История заказов ----

Покупатель видит только свои оформленные заказы.

История не показывает `basket`.

Список заказов должен возвращать:

- ID заказа;
- дату создания;
- дату подтверждения;
- статус;
- сумму по снимкам;
- количество позиций.

## 14.2. Детали заказа ----

Детали заказа должны возвращать:

- ID заказа;
- статус;
- контакт доставки;
- позиции;
- магазин по каждой позиции;
- ценовой снимок;
- количество;
- сумму позиции;
- общую сумму;
- причины отмены, если есть.

Покупатель видит только свои заказы.

Поставщик не получает этот endpoint как смешанный заказ. Для поставщика есть
специальный список его `OrderItem`.

## 14.3. Поставщик и позиции заказов ----

Поставщик работает только с позициями, где `product_info.shop` является его
магазином.

Поставщик видит:

- ID позиции;
- ID заказа или внешний номер заказа;
- название товара или snapshot;
- модель;
- количество;
- цену, если она нужна для исполнения;
- статус позиции;
- минимальные данные доставки, если это требуется бизнес-сценарием.

Поставщик не видит:

- позиции других магазинов;
- полный состав смешанного заказа;
- лишние персональные данные покупателя.

## 15. Административный API ----

Административные endpoint'ы доступны только пользователям:

- `type="admin"`;
- `is_staff=True`;
- `is_active=True`.

Администратор управляет:

- пользователями;
- магазинами;
- категориями;
- товарами;
- параметрами;
- предложениями;
- заказами;
- позициями заказов.

Административные операции должны использовать сервисы, а не прямую запись из
view в модель.

Особые правила:

- отмена заказа требует причину;
- нельзя сделать `admin` без `is_staff=True`;
- нельзя сделать `shop` без связанного магазина;
- нельзя менять роль владельца магазина на `buyer`;
- нельзя уменьшить `Offer.quantity` ниже `reserved_quantity`;
- нельзя нарушать складские ограничения при изменении статуса позиции.

## 17. DB-инварианты и ограничения ----

Обязательные текущие или целевые ограничения:

- `User.email` уникален;
- `ProductInfo.quantity >= 0`;
- `ProductInfo.reserved_quantity >= 0`;
- `ProductInfo.reserved_quantity <= ProductInfo.quantity`;
- `ProductInfo.price > 0`;
- `ProductInfo.price_rrc >= 0`;
- `OrderItem.quantity > 0`;
- `OrderItem(order, product_info)` уникален;
- `ProductParameter(product_info, parameter)` уникален;
- `Parameter.name` уникален после нормализации;
- `ProductInfo(shop, external_id)` уникален, если `external_id` заполнен;
- у пользователя не должно быть больше одной открытой корзины;
- `Shop.owner` уникален.

Если ограничение трудно выразить на уровне БД, оно должно быть явно покрыто
транзакционным сервисом и тестами.

## 18. Ошибки API ----

Рекомендуемые статусы:

- `400 Bad Request` - неверные данные запроса или нарушение бизнес-правила;
- `401 Unauthorized` - пользователь не авторизован;
- `403 Forbidden` - роль не имеет права выполнять действие;
- `404 Not Found` - объект не найден или не принадлежит пользователю;
- `409 Conflict` - конфликт состояния или конкурентное изменение;
- `500 Internal Server Error` - непредвиденная ошибка.

Примеры бизнес-ошибок:

- количество меньше 1;
- запрошенное количество больше доступного остатка;
- подтверждение пустой корзины;
- подтверждение с чужим контактом;
- попытка купить скрытое предложение;
- попытка поставщика изменить чужое предложение;
- попытка поставщика увидеть чужую позицию;
- попытка покупателя отменить заказ после начала обработки;
- попытка администратора отменить заказ без причины;
- попытка снизить остаток ниже резерва.

## 19. Логирование ----

Можно логировать:

- ID пользователя;
- ID магазина;
- ID заказа;
- ID позиции;
- ID предложения;
- факт запуска импорта;
- результат импорта;
- количество обработанных строк;
- изменение статуса;
- технические ошибки;
- длительность операции.

Нельзя логировать:

- пароли;
- JWT;
- cookies;
- секретные ключи;
- полный email в лишних местах;
- телефон;
- адрес доставки;
- полное тело запроса с персональными данными;
- содержимое файлов с персональными данными.

Формат сообщений в коде:

```python
logger.info("[function_name] что происходит")
```

В текущем коде уже используется структурный формат с ID объектов. При новых
изменениях нужно придерживаться единого стиля и не добавлять персональные данные.

## 20. Email и уведомления ----

Базовые уведомления:

- письмо покупателю после подтверждения заказа;
- письмо администраторам после подтверждения заказа;
- письмо активации пользователя, если включена Djoser-активация.

Дополнительные уведомления:

- поставщику о новой позиции;
- покупателю об изменении статуса;
- покупателю об отмене;
- администратору об ошибке импорта.

Правила:

- ошибка email не откатывает оформленный заказ;
- отправка должна запускаться после commit;
- медленную отправку нужно вынести в Celery;
- email-ошибки логируются без персональных данных.

## 21. Асинхронные задачи ----

Celery относится к продвинутой части, но архитектура должна быть готова к нему.

Кандидаты на задачи:

- `send_order_confirmation_email`;
- `send_order_status_changed_email`;
- `import_shop_price`;
- `export_shop_offers`;
- `cleanup_expired_baskets`;
- `recalculate_catalog_indexes`.

Сначала нужно стабилизировать синхронные сервисы и тесты. После этого можно
выносить тяжелые операции в фон без изменения бизнес-контрактов.

## 22. Главный пользовательский сценарий ----

К финальной версии должен работать сценарий:

1. Покупатель регистрируется через API.
2. Покупатель активирует аккаунт, если включена активация.
3. Покупатель получает JWT.
4. Поставщик регистрирует магазин.
5. Администратор одобряет магазин.
6. Поставщик создает предложения или импортирует YAML-прайс.
7. Покупатель открывает каталог.
8. Покупатель выбирает конкретные предложения магазинов.
9. Покупатель добавляет предложения в корзину.
10. Покупатель создает контакт доставки.
11. Покупатель подтверждает корзину.
12. Система резервирует остатки и фиксирует цены.
13. Покупатель получает подтверждение.
14. Поставщики видят свои позиции.
15. Поставщик переводит позицию по статусам.
16. При отправке позиция списывает физический остаток.
17. Заказ пересчитывает общий статус.
18. Покупатель видит историю и детали заказа.

## 23. Что считается готовой базовой частью ----

Базовая часть считается готовой, если:

- проект запускается в Docker;
- миграции применяются;
- регистрация покупателя работает;
- JWT-авторизация работает;
- регистрация магазина работает;
- администратор может одобрить магазин;
- магазин может управлять своими предложениями;
- каталог показывает доступные к покупке предложения;
- фильтры применяются корректно;
- корзина возвращает позиции и суммы;
- добавление, изменение и удаление позиции корзины работает;
- подтверждение заказа атомарно резервирует остатки;
- заказ хранит ценовой снимок;
- контакты работают с soft-delete;
- покупатель видит только свои заказы;
- поставщик видит только свои позиции;
- статусы позиций двигаются по автомату;
- статус заказа пересчитывается по позициям;
- административная отмена требует причину;
- OpenAPI не содержит предупреждений;
- ключевые бизнес-инварианты покрыты тестами.

## 26. Тестовая матрица ----

## 26.1. Auth и роли ----

- покупатель регистрируется через Djoser;
- обычная регистрация не позволяет создать `shop` или `admin`;
- неактивный пользователь не получает JWT;
- магазин регистрируется через `/shops/register/`;
- admin endpoint'ы доступны только `type=admin` и `is_staff=True`;
- buyer не имеет доступа к shop/admin endpoint'ам;
- shop не имеет доступа к buyer basket/contact endpoint'ам.

## 26.2. Каталог ----

- неавторизованный пользователь получает `401`, пока каталог закрыт;
- активный покупатель видит только активные предложения активных магазинов;
- архивная категория скрывает предложения;
- архивный товар скрывает предложения;
- hidden/blocked/archived offer не доступен к покупке;
- фильтры по магазину и цене применяются к одному предложению;
- фильтры по магазину и характеристике применяются к одному предложению;
- предложение возвращает доступный остаток с учетом резерва.

## 26.3. Корзина ----

- пустая корзина возвращает корректный объект;
- первое добавление создает корзину;
- повторное добавление того же offer увеличивает quantity;
- два offer одного Product от разных магазинов создают разные позиции;
- нельзя добавить 0 или отрицательное количество;
- нельзя добавить больше доступного остатка;
- нельзя добавить скрытое предложение;
- нельзя удалить чужую позицию;
- нельзя изменить позицию оформленного заказа;
- конкурентное добавление не создает дубли `OrderItem`.

## 26.4. Контакты ----

- покупатель создает контакт;
- покупатель видит только свои контакты;
- чужой контакт возвращает `404`;
- удаление неиспользованного контакта физически удаляет его;
- удаление использованного контакта выставляет `is_deleted=True`;
- удаленный контакт нельзя использовать при подтверждении заказа.

## 26.5. Заказы ----

- нельзя подтвердить пустую корзину;
- нельзя подтвердить корзину с чужим контактом;
- подтверждение резервирует остатки;
- подтверждение фиксирует цену;
- изменение цены предложения после подтверждения не меняет сумму заказа;
- ошибка email не откатывает заказ;
- покупатель видит только свои заказы;
- `basket` не попадает в историю;
- покупатель может отменить заказ до обработки;
- покупатель не может отменить заказ после `accepted`;
- отмена освобождает резерв.

## 26.6. Поставщик ----

- поставщик видит только свои предложения;
- поставщик не может обновить чужое предложение;
- pending/blocked shop не может создавать активные предложения;
- поставщик видит только свои позиции заказов;
- поставщик переводит позицию только вперед;
- перевод в `sent` списывает остаток;
- отмена до отправки освобождает резерв;
- отмена после отправки запрещена.

## 26.7. Администрирование ----

- администратор видит справочники;
- администратор может архивировать категорию и товар;
- администратор может блокировать предложение;
- администратор не может отменить заказ без причины;
- административная отмена освобождает резерв;
- нельзя сделать admin без `is_staff`;
- нельзя сделать shop без связанного магазина;
- нельзя изменить роль владельца магазина на buyer.

## 26.8. Импорт ----

- shop запускает импорт своего прайса;
- buyer не может импортировать прайс;
- неавторизованный пользователь не может импортировать прайс;
- повторный импорт с тем же `external_id` обновляет offer;
- изменение `name` при том же `external_id` не создает дубль;
- невалидный YAML откатывает импорт;
- неизвестная категория в API-импорте откатывает импорт;
- цена `<= 0` откатывает импорт;
- отрицательный остаток откатывает импорт;
- отсутствующие в новом прайсе предложения скрываются.

## 27. Документация и OpenAPI ----

Документация должна отражать реальную реализацию.

Обязательные правила:

- после изменения API обновлять serializers и `api/openapi.py`;
- Swagger должен объяснять различие `Product` и `Offer`;
- legacy endpoint'ы должны быть помечены как временные;
- README должен описывать только актуальные команды запуска и проверки;
- `.env.example` и `.env` должны синхронно обновляться при новых переменных.

## 28. Итоговые бизнес-инварианты ----

Эти правила должны соблюдаться всегда:

- пользователь не видит чужие объекты;
- покупатель работает только со своей корзиной, заказами и контактами;
- поставщик работает только со своим магазином, предложениями и позициями;
- администратор действует через сервисы и не нарушает складские ограничения;
- `Product` не содержит цену и остаток;
- `ProductInfo/Offer` содержит цену, остаток, магазин и характеристики;
- в корзину добавляется только `Offer`;
- один `Offer` в корзине не дублируется;
- количество позиции всегда больше нуля;
- доступный остаток учитывает резерв;
- подтверждение заказа атомарно;
- подтверждение резервирует остаток;
- отправка списывает физический остаток;
- отмена до отправки освобождает резерв;
- заказ хранит ценовой снимок;
- история заказа не меняется после изменения предложения;
- поставщик не видит чужие позиции;
- магазин не получает новые заказы, если он не активен или не принимает заказы;
- email-ошибка не ломает заказ;
- персональные данные не логируются без необходимости;
- все спорные бизнес-правила закрепляются тестами.
