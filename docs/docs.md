# 1. Архитектура документации ----

Документация проекта должна отражать фактическую реализацию API и доменной
логики. Если поведение меняется в коде, связанный документ обновляется в том же
изменении.

## 1.1. Обязательная структура ----

- `docs/docs.md` - карта документации и правила ее поддержки.
- `docs/env.md` - все переменные окружения, значения по умолчанию и способ
  получения.
- `docs/TEST.md` - тестовая матрица бизнес-логики из `AGENTS_SPEC.md`.
- `docs/api.md` - актуальные endpoint'ы, роли и переходные legacy-контракты.
- `docs/django.md` - техническая памятка по Django, Makefile, Docker и логгингу.

## 1.2. Правила обновления ----

- При изменении публичного API обновлять `api/serializers.py`, `api/openapi.py`
  и `docs/api.md`.
- При добавлении env-переменной обновлять `.env.example`, `.env` без изменения
  существующих значений и `docs/env.md`.
- При изменении бизнес-инварианта обновлять `AGENTS_SPEC.md`, тесты и
  `docs/TEST.md`.
- Legacy endpoint'ы документировать как временные, пока они остаются в коде.

## 1.3. Что документировать отдельно ----

- Доменную модель: роли, товары `Product`, предложения `Offer/ProductInfo`,
  корзина, заказы, контакты и импорт.
- Складские инварианты: резерв, списание при `sent`, отмена до отправки.
- Права доступа: `buyer`, `shop`, `admin`, ответы `403` и `404` для чужих
  объектов.
- Операционные команды: Docker, миграции, тесты, OpenAPI validation,
  pre-commit.
