# Техническое задание — Procurement API (Сервис закупок)

> **Проект:** `django-procurement-api`
> **Версия:** 4.1
> **Дата:** 2026-06-26
> **Статус:** Финальная версия
> **Описание:** REST API платформы для взаимодействия покупателей, магазинов и администраторов в сфере электронных закупок.

---

## 1. Назначение системы

REST API платформы электронных закупок автоматизирует полный цикл работы трёх ролей:

- **Покупатель** ищет товары в каталоге, собирает корзину, оформляет и отслеживает заказы.
- **Магазин** регистрируется на платформе, загружает прайс-листы в формате YAML, управляет своими предложениями и обрабатывает поступившие заказы.
- **Администратор** модерирует магазины, наполняет справочники (категории, товары, характеристики) и управляет заказами с расширенными правами.

Система построена на Django 5.2 и Django REST Framework 3.17 с асинхронной обработкой фоновых задач через RQ (Redis Queue), JWT-аутентификацией через Djoser и SimpleJWT, автоматической OpenAPI-документацией через drf-spectacular.

## 2. Ключевые бизнес-правила

### 2.1. Пользователи и роли

Единая модель `User` авторизуется по email. Тип пользователя задаётся полем `type`: `buyer`, `shop` или `admin`. Обычная регистрация создаёт покупателя; магазин регистрируется через отдельный endpoint `/api/shops/register/`. Новый пользователь создаётся **неактивным** — требуется активация через email или действие администратора.

Администратор — это пользователь с `is_staff=True` и `type=admin`. Только администратор может активировать магазин, перевести его в статус `active` и управлять справочниками.

### 2.2. Магазины и модерация

Регистрация магазина создаёт пользователя (`type=shop`, `is_active=False`) и запись магазина в статусе `pending`. Администратор одобряет (`active`) или блокирует (`blocked`) магазин. Только активный магазин может создавать предложения, импортировать прайсы и принимать заказы.

Блокировка не отменяет уже принятые заказы — магазин продолжает их обрабатывать, но новые заказы не принимает.

### 2.3. Каталог: товары и предложения

Система разделяет абстрактный товар (`Product` — единица каталога) и конкретное предложение магазина (`ProductInfo` или `Offer` — цена, остаток, характеристики). Покупатель видит только активные товары и предложения от активных магазинов, принимающих заказы.

Категории кэшируются для ускорения загрузки каталога. Кэш автоматически сбрасывается при любых изменениях категорий, товаров или предложений через Django-сигналы.

### 2.4. Корзина и заказы

Корзина — это заказ в статусе `basket`. У пользователя может быть только одна активная корзина. Повторное добавление того же предложения увеличивает количество. Система проверяет доступный остаток с учётом уже зарезервированного и использует row-level блокировки (`SELECT ... FOR UPDATE`) для предотвращения race conditions.

Подтверждение заказа переводит его в статус `confirmed`, резервирует остаток по всем позициям, сохраняет снепшоты цен и названий, и через RQ-воркер отправляет email-уведомления покупателю и администраторам.

Управление заказами — это статусная машина:

```
basket → confirmed → processing → sent → delivered
                          ↓
                      canceled ← partially_canceled
```

Покупатель может отменить заказ только до начала обработки магазином. Отмена освобождает резерв. Отправка списывает товар со склада. Магазин управляет позициями линейно (принял → собрал → отправил → доставил) с возможностью отмены. Администратор имеет расширенную таблицу переходов с возможностью перепрыгивать через статусы.

### 2.5. Импорт прайс-листов

Магазин загружает прайс-лист в формате YAML — файлом через multipart или строкой в JSON. Валидация выполняется fail-fast перед началом импорта. Система атомарно обновляет предложения по ключу `(shop, external_id)`, создаёт недостающие категории, товары и предложения, заменяет характеристики. Предложения, отсутствующие в новом прайсе, автоматически скрываются (`status=hidden`).

Импорт в API запускается асинхронно. Статус выполнения отслеживается через `ImportJob`. При ошибке транзакция откатывается, а запись импорта фиксируется как `failed` с текстом ошибки.

### 2.6. Контакты и soft-delete

Адрес доставки принадлежит пользователю. При удалении контакта, если он используется в оформленных заказах (не корзинах), выполняется soft-delete: контакт помечается `is_deleted=True`, но остаётся в базе для целостности истории заказов.

### 2.7. Защита от конкурентного доступа

Все операции, изменяющие остатки на складе — добавление в корзину, обновление количества, подтверждение заказа, отмена, отгрузка — используют `select_for_update()` внутри транзакций. Это гарантирует, что два параллельных запроса не продадут один и тот же товар дважды.

## 3. API endpoints

Базовый URL: `/api/`.

**Без аутентификации:**
- `GET /api/health/` — проверка работоспособности сервиса и БД
- `POST /api/auth/users/` — регистрация покупателя
- `POST /api/auth/users/activation/` — активация по uid/token
- `POST /api/auth/jwt/create/` — получение JWT
- `POST /api/auth/jwt/refresh/` — обновление токена
- `POST /api/auth/jwt/verify/` — проверка токена
- `POST /api/auth/users/reset_password/` — сброс пароля
- `POST /api/auth/users/reset_password_confirm/` — подтверждение сброса
- `POST /api/shops/register/` — регистрация магазина

**С JWT (роль Buyer):**
- `GET /api/categories/` — активные категории
- `GET /api/products/` — товары (с фильтрацией)
- `GET /api/offers/` — предложения (с фильтрацией)
- `GET /api/products/{id}/offers/` — предложения конкретного товара
- `GET /api/offers/{id}/` — карточка предложения
- `GET/DELETE /api/basket/` — просмотр/очистка корзины
- `POST /api/basket/items/` — добавить в корзину
- `PATCH/DELETE /api/basket/items/{id}/` — изменить/удалить позицию корзины
- `GET/POST /api/contacts/` — список/создание контактов
- `GET/PATCH/DELETE /api/contacts/{id}/` — управление контактом
- `POST /api/order/confirm/` — подтверждение заказа
- `GET /api/orders/` — история заказов
- `GET /api/orders/{id}/` — детали заказа
- `POST /api/orders/{id}/cancel/` — отмена заказа

**С JWT (роль Shop):**
- `GET/PATCH /api/shop/profile/` — профиль магазина
- `GET/POST /api/shop/offers/` — предложения магазина
- `PATCH /api/shop/offers/{id}/` — обновление предложения
- `POST /api/shop/imports/` — импорт YAML-прайса
- `GET /api/shop/imports/{id}/status/` — статус импорта
- `GET /api/shop/order-items/` — позиции заказов магазина
- `PATCH /api/shop/order-items/{id}/` — изменение статуса позиции

**С JWT (роль Admin):**
- `GET /api/auth/users/me/` — профиль текущего пользователя
- `GET/PATCH /api/admin/users/` и `/api/admin/users/{id}/` — управление пользователями
- `GET/PATCH /api/admin/shops/` и `/api/admin/shops/{id}/` — управление магазинами
- `POST /api/admin/shops/{id}/approve/` и `.../block/` — модерация
- `GET/POST/PATCH /api/admin/categories/` — справочник категорий
- `GET/POST/PATCH /api/admin/products/` — справочник товаров
- `GET/POST/PATCH /api/admin/parameters/` — справочник характеристик
- `GET/PATCH /api/admin/offers/` — все предложения
- `GET/PATCH /api/admin/orders/` — все заказы
- `PATCH /api/admin/order-items/{id}/` — любая позиция заказа

### 3.1. Фильтрация каталога

**Товары** фильтруются по: `search` (название), `category_id`, `shop_id`, `price_min`/`price_max`, `parameter` (формат `имя_параметра:значение`).

**Предложения** дополнительно поддерживают: `in_stock` (только в наличии), `ordering` (по `id`, `price`, `quantity`).

## 4. Модели данных

Система включает 11 моделей:

- **User** — покупатель/магазин/администратор; авторизация по email, тип задаётся полем `type`.
- **Shop** — магазин; привязан к User через `OneToOneField`. Статусы: `pending → active | blocked | archived`.
- **Category** — категория товаров; ManyToMany с Shop. Статусы: `active | archived`.
- **Product** — товар каталога; принадлежит категории. Статусы: `active | archived`.
- **ProductInfo (Offer)** — предложение магазина с ценой, количеством и резервом. Статусы: `active | hidden | archived | blocked`. Уникальность: `(shop, external_id)`.
- **ProductParameter** — характеристика предложения; связь ProductInfo → Parameter.
- **Parameter** — справочник характеристик.
- **Contact** — адрес доставки покупателя. Поддерживает soft-delete.
- **Order** — заказ; статусная машина из 7 состояний. Уникальный basket на пользователя.
- **OrderItem** — позиция заказа; хранит снепшоты цен и названий на момент подтверждения.
- **ImportJob** — задача импорта прайса; статусы: `pending → processing → completed | failed`.

### 4.1. Целостность данных

Все ключевые ограничения заданы на уровне базы данных через `CheckConstraint` и `UniqueConstraint`:

- `quantity >= 0`, `reserved_quantity >= 0` и `<= quantity`, `price > 0`, `price_rrc >= 0` (ProductInfo)
- `quantity >= 1` (OrderItem)
- Уникальный active basket на пользователя (`state=basket`)
- Уникальная связка `(shop, external_id)` при непустом external_id
- Уникальная связка `(product_info, parameter)` для характеристик
- Уникальная связка `(order, product_info)` для позиций заказа

## 5. Инфраструктура

### 5.1. Docker-окружение

Пять контейнеров:

| Сервис | Назначение |
|--------|------------|
| `db` | PostgreSQL 16 — основная БД |
| `redis` | Redis 7 — кэш и брокер RQ |
| `web` | Python 3.10 + Django — API, Gunicorn/runserver |
| `worker` | RQ Worker — асинхронная отправка email |
| `test-db` | PostgreSQL 16 — одноразовая БД для тестов (профиль test) |

### 5.2. Кэширование

В разработке — `LocMemCache`. В production — Redis. Кэшируются: список активных категорий, детальные карточки предложений. Время жизни — 5 минут. Сброс кэша — автоматический через сигналы при изменении категорий, товаров, предложений.

### 5.3. Асинхронная обработка (RQ)

RQ Worker обрабатывает задачи из очереди `default`. Единственная задача на текущий момент — отправка email-подтверждения заказа. RQ Dashboard доступен по `/django-rq/`.

### 5.4. Rate Limiting

| Scope | Лимит | Применение |
|-------|-------|------------|
| `anon` | 20 запросов/мин | Анонимные запросы |
| `user` | 60 запросов/мин | Аутентифицированные пользователи |
| `shop_register` | 5 запросов/мин | Регистрация магазинов |

### 5.5. Логирование

`RequestLogMiddleware` логирует каждый HTTP-запрос с методом, путём, статус-кодом, длительностью и ID пользователя. Уровень зависит от кода ответа: `INFO` (2xx), `WARNING` (4xx), `ERROR` (5xx). Сервисы логируют бизнес-операции в формате: `[название_функции] событие key=value key2=value2`.

Логирование управляется через `.env`: `DJANGO_LOGGING_ENABLED`, `DJANGO_LOG_LEVEL`, `DJANGO_LOG_MODULES`, `DJANGO_LOG_SQL`.

## 9. Архитектурные решения и их обоснование

**Почему RQ для email, а не threading.Thread?**
RQ даёт повторяемость (worker перезапускает упавшие задачи), мониторинг (Web UI), graceful shutdown и единый подход для всех асинхронных задач в будущем. Импорт прайса пока выполняется в потоке по историческим причинам — в следующей итерации планируется перевод на RQ.

**Почему row-level блокировки вместо оптимистичных?**
Бизнес-операции (корзина, подтверждение заказа) короткие и работают с конкретными записями. `SELECT FOR UPDATE` даёт строгую сериализацию без retry-логики на стороне приложения.

**Почему разделение Product и ProductInfo?**
Один товар (iPhone XS) может продаваться в разных магазинах по разным ценам. Product — это единица каталога, ProductInfo — конкретное предложение магазина с ценой, остатком и характеристиками. Это упрощает модерацию каталога администратором и даёт магазинам гибкость в ценообразовании.

**Почему мягкое удаление (soft-delete) для контактов?**

Контакты используются в заказах. Физическое удаление контакта, на который уже оформлен заказ, нарушит целостность истории. Soft-delete сохраняет данные для архива, но скрывает удалённый контакт от пользователя в интерфейсе выбора адреса.

## 10. Планируемые доработки (следующая итерация)

По результатам ревью проекта шестью специалистами (архитектор, product owner, QA, devops, security, разработчик) сформирован список улучшений на следующий релиз:

1. **Перевод импорта на RQ** — единый механизм асинхронной обработки, graceful shutdown, мониторинг.
2. **Уведомления магазину** — email при появлении новых позиций в заказах магазина.
3. **Оптимизация N+1** — исправление `BuyerOfferSerializer.get_parameters`.
4. **Фильтрация заказов по статусу** — для админ-панели.
5. **Тесты на сигналы кэширования** — проверка инвалидации при изменениях.
6. **Синхронизация лимитов YAML** — единый порог (5 MB) в настройках и сериализаторе.
7. **Унификация пагинации** — единый DEFAULT_PAGE_SIZE.
8. **Базовый класс для валидации offer** — устранение дублирования проверок `price > 0`, `quantity >= 0`.

## 11. Заключение

Procurement API — это законченный REST-сервис электронных закупок с трёхролевой моделью, JWT-аутентификацией, импортом YAML-прайсов, статусной машиной заказов, защитой от race conditions, кэшированием, асинхронными уведомлениями, OpenAPI-документацией и тестовым покрытием более 85%. Система готова к развёртыванию в Docker-окружении и может быть расширена без изменения основной архитектуры.
