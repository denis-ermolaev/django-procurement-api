# 1. Тестовая матрица ----

Базовая проверка после изменений:

```bash
make compose_without_build
make test
make schema_validate
make pre-commit_host
```

`make test` запускает Django tests в web-контейнере на PostgreSQL.

## 1.1. Auth и роли ----

- Регистрация через Djoser создает только `buyer`.
- Попытка создать `shop` или `admin` через обычную регистрацию возвращает `400`.
- Неактивный пользователь не получает JWT.
- `buyer`, `shop`, `admin` имеют раздельные permissions.
- Неактивный пользователь не проходит ролевые permissions.
- Admin endpoints доступны только `type=admin`, `is_staff=True`, `is_active=True`.

Покрытие: `api/tests/test_auth.py`, `api/tests/test_shops.py`,
`api/tests/test_role_api.py`.

## 1.2. Каталог и предложения ----

- Неавторизованный пользователь получает `401`.
- `Product` не содержит цену и остаток.
- `Offer/ProductInfo` содержит магазин, цену, остаток, модель и характеристики.
- `/api/offers/`, `/api/offers/{id}/`, `/api/products/{id}/offers/` показывают
  только активные предложения активных магазинов, которые принимают заказы.
- Фильтры `shop_id`, `price_min`, `price_max`, `parameter`, `in_stock` применяются
  к одному и тому же предложению.
- Hidden/blocked/archived offer не доступен к покупке.

Покрытие: `api/tests/test_products.py`.

## 1.3. Корзина ----

- Пустая корзина возвращает объект `{id, state, items, total}`.
- Первое добавление создает basket-заказ.
- Повторное добавление того же offer увеличивает `quantity`.
- `/api/basket/items/{id}/` изменяет и удаляет только позицию текущей корзины.
- Нельзя добавить или установить quantity больше доступного остатка.
- Один пользователь имеет одну активную корзину.
- Один offer не дублируется внутри заказа.

Покрытие: `api/tests/test_basket.py`, `api/tests/test_models.py`.

## 1.4. Контакты ----

- Покупатель создает и видит только свои контакты.
- Чужой или soft-deleted контакт возвращает `404`.
- Неиспользованный контакт удаляется физически.
- Использованный контакт получает `is_deleted=True`.
- `/api/contacts/{id}/` поддерживает `GET`, `PATCH`, `DELETE`.

Покрытие: `api/tests/test_contacts.py`.

## 1.5. Заказы и склад ----

- Нельзя подтвердить пустую корзину.
- Нельзя подтвердить заказ с чужим или удаленным контактом.
- Подтверждение атомарно резервирует остаток.
- Подтверждение фиксирует `unit_price`, `price_rrc_snapshot`, названия товара,
  предложения, магазина и `external_id`.
- Изменение цены offer после подтверждения не меняет историю заказа.
- Email отправляется через `transaction.on_commit`.
- Отмена до отправки освобождает резерв.
- Перевод позиции в `sent` списывает резерв и физический остаток.
- Поставщик двигает только свои позиции и только вперед.
- Pending/blocked/paused shop не обрабатывает позиции.

Покрытие: `api/tests/test_orders.py`, `api/tests/test_role_api.py`.

## 1.6. Администрирование ----

- Администратор управляет пользователями, магазинами, справочниками и
  предложениями через сервисы.
- Административная отмена требует `cancellation_reason`.
- Административная отмена до отправки освобождает резерв.
- Отправленные и доставленные позиции нельзя вернуть к досылочным статусам или
  отменить без отдельной возвратной операции.
- Нельзя назначить `admin` без `is_staff=True`, `shop` без магазина и снять роль
  владельца магазина.

Покрытие: `api/tests/test_role_api.py`, `api/tests/test_models.py`.

## 1.7. Импорт ----

- Management command `load_shop_data` идемпотентна для demo YAML.
- `goods[].id` сохраняется в `ProductInfo.external_id`.
- `goods[].model` сохраняется в `ProductInfo.model`.
- Повторный импорт с тем же `external_id` обновляет offer и не создает дубль.
- API `/api/shop/imports/` доступен только активному магазину.
- Невалидный YAML, неизвестная категория, цена `<= 0` и отрицательный остаток
  возвращают `400` без частичного изменения базы.
- Предложения с `external_id`, отсутствующие в новом прайсе, скрываются.

Покрытие: `api/tests/test_load_shop_data.py`.

## 1.8. OpenAPI и логгинг ----

- Swagger и schema доступны публично.
- `make schema_validate` проходит без warnings.
- Request logging не пишет тело запроса, JWT, cookies и персональные данные.
- Бизнес-логи включаются только через `.env`.
- Сообщения логов используют формат `[function_name] событие`.

Покрытие: `api/tests/test_schema.py`, `api/tests/test_logging.py`.
