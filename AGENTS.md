# 1. Запуск проекта ----
* Запуск контейнера - `make compose_without_build` (предпочтительным способом запуска проекта, так как позволяет экономить интернет-трафик и время).
* Следует по максимуму используй контейризацию, чтобы иметь унифицированные условия и правильное окружение, например для просмотр ошибок Django, выполнение management-команд, запуска тестов, проверки функционала, логов и т.п.

Если для выполнения задачи требуется пересборка контейнеров:

* останови выполнение задачи;
* сообщи пользователю, что для проверки изменений требуется пересборка проекта;
* не пытайся обходить проблему самостоятельно.

# 2. Проверки после изменений ----
```bash
# После каждого изменения обязательно запускай:
make pre-commit_host
# Тесты:
make test
```

Если возникают ошибки `ty`, `mypy`, не пройденные тесты, то исправляй их, если они вызваны текущими изменениями.

# 3. Workflow ----
1. Изучи файлы `AGENTS.md` (Инструкции для работы) и `AGENTS_SPEC.md` (Пояснение как должно работать) в корневой директории, а также в папке, где предстоит работа, если они там есть.
2. Тщательно изучи код, необходимый для решения задачи.
3. Сформируй план работы
4. Внеси изменения
5. Запусти проверки после изменений, исправь найденные ошибки, повтори проверку
6. Измени файлы документации `/docs/`, если необходимо, чтобы они всегда были про актуальную реализацию.

Используй следующие инструкции для работы:

- Не выполняй крупный рефакторинг без необходимости.
- Используй по-максимуму Makefile, сохраняй туда важные команды, которые предполагается часто использовать.
- Покрывай новый код тестами, в соответствии с бизнес логикой. Ты даже можешь в начале писать тесты, а после реализовывать новую фичу. После чего попробуй покрыть лучше тестами, крайние значения, чтобы всё было точно стабильно.
- Используй логгинг и комментарии на русском языке
- Никогда без разрешения не изменяй файлы `README.md`, `AGENTS.md`, `AGENTS_check.md`
- Тесты запускай сразу же на PostgreSQL, чтобы иметь максимально приближенное к реальности окружение.

# 4. Переменные окружения (env) ----

Оставляй настройки в файле `.env`, но держи его структуру в соответствии с `.env.example` при каких-то ошибках или несоответсвии, сообщай пользвателю. При добавлении чего-то в `.env.example`, добавляй это сразу в `.env`.

# 5. Комментарии и logging ----

Для Makefile
используем следующие виды комметариев:
```
### Название группы команд, например внутри контейнера, БД и т.п
## Название конкретной одиночной команды

Например:

### Проверки качества
## Запустить тесты, покрытие, OpenAPI validation и pre-commit
check_host: coverage_host schema_validate_host pre-commit_host

## Запустить pre-commit хуки
pre-commit_host:
	uv run pre-commit run --all-files
```

В Python файлах комментарии
```
# 1. Заголовок первого уровня ----

## 1.1. Заголовок второго уровня ----

```

Во всех файлах можно выделять крупные блоки для удобного скрытия кода
```
Большие блоки

# ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ #
#                            УПРАВЛЕНИЕ КОНТЕЙНЕРОМ                            #
# ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ #

Тут прочие команды

Или код


# ---------------------------------------------------------------------------- #

```

Добавляй комментарии в сложные места, чтобы объяснять код или логику.

Для логирования используй формат "[название функции] что происходит"

# 6. Итоговый отчёт ----

## 6.1. Ход работы ----
Кратко опиши ход работы, чтобы было в результате сделано, не забывай работать внутри докер контейнра, там где это возможно.

Подчёркивай моменты, которые особенно необходимо выладировать человеку, архитектурные решения, бизнес-логика.


## 6.2 Проверка ----
Обязательно говори результаты тестов, проверок и т.п


## 6.3 Предложение для коммита ----
Всегда предлагай готовое название коммита в формате `type(scope): краткое и конкретное описание, того, что было сделано`:
```text
feat(views-ProductListView): фильтрация
feat(auth): система авторизации
fix(api): корзина и детали заказов
docs(api): улучшение swagger и документации API
build: настройка окружения: линтеры, тайпчекеры, форматтеры и pre-commit хуки
test(api): добавлены основные тесты заказов
fix(shop-data): видимость предустановленного каталога из yaml-файлов
```


Коммиты пиши на русском языке, кроме типа действия: fix, feat, refactor, build, test, docs, также в скобках можешь указывать области действия:

- (views-ProductListView) - изменение в файле views, в классе ProductListView
- (basket) - изменения в нескольких файлах views, models, serializers связанных с реализацией корзины
- (shop-data) - отновиться к реализации shop, но конкретно к папки data
- (auth) - если работа идёт над авторизацией
- (api) - коммит затрагивает много файлов во всём приложении
