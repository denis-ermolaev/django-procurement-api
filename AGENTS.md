# AGENTS.md

## Общие правила работы

* Перед внесением изменений сначала изучи связанные файлы и текущую реализацию.
* Предпочитай минимальные и точечные изменения вместо крупных рефакторингов.
* Не изменяй код, не относящийся к поставленной задаче.
* Не изменяй публичные интерфейсы без явного требования.
* Не удаляй и не ослабляй тесты ради прохождения проверок.
* Не выполняй коммит без явного указания пользователя.
* Если есть несколько вариантов решения, выбирай наиболее простой и наименее рискованный.

## Работа с Docker

* Используй `docker-compose.yml` проекта.
* Для запуска проекта используй:

```bash
make compose_without_build
```

* Не выполняй пересборку контейнеров без явного разрешения пользователя.
* Не используй `docker compose build`, `docker compose up --build` и аналогичные команды без необходимости.
* Команда `make compose_without_build` является предпочтительным способом запуска проекта, так как позволяет экономить интернет-трафик и время. По максимуму используй контейризацию, чтобы иметь унифицированные условия и правильное окружение!

Если для выполнения задачи требуется пересборка контейнеров:

* останови выполнение задачи;
* сообщи пользователю, что для проверки изменений требуется пересборка проекта;
* не пытайся обходить проблему самостоятельно.

## Использование контейнеров

По возможности используй окружение контейнеров вместо хостовой системы.

Например:

* просмотр ошибок Django;
* выполнение management-команд;
* запуск тестов;
* проверка работы backend-функционала;
* просмотр логов сервисов.

Предпочитай проверку поведения приложения внутри контейнера, если это позволяет получить более достоверный результат.

## Проверки после изменений

После каждого изменения обязательно запускай:

```bash
make pre-commit_host
```

Если изменяется backend-логика:

* запускай минимально необходимый набор тестов;
* если затронута большая область функциональности, запускай:

```bash
make test_host
```

Если возникают ошибки:

* `ty`
* `mypy`

то исправляй их, если они вызваны текущими изменениями.

## Работа с ошибками

* Внимательно анализируй вывод команд перед внесением следующих изменений.
* Исправляй только ошибки, относящиеся к текущей задаче.
* Если обнаружена старая или сторонняя проблема проекта, не выполняй масштабный рефакторинг для её обхода.
* Укажи такую проблему отдельно в итоговом отчёте.

## Django

* Для анализа ошибок используй реальные логи и сообщения Django.
* Не делай предположений о причине ошибки без проверки.
* Не создавай миграции без необходимости.
* Если задача требует создания миграций, явно укажи это в итоговом отчёте.

## Итеративная работа

Работай циклически:

1. Изучи код.
2. Сформулируй предполагаемую причину проблемы.
3. Внеси минимальное изменение.
4. Запусти релевантную проверку.
5. Исправь найденные ошибки.
6. Повтори проверку.
7. Только после успешного прохождения локальной проверки переходи к более широким тестам.

Если проверка завершилась ошибкой, не считай задачу выполненной.
Сначала попытайся исправить проблему, затем повторно запусти проверку.

Не выполняй крупный рефакторинг без необходимости.

## env

Оставляй настройки в файле `.env`, но держи его структуру в соответствии с `.env.example` при каких-то ошибках с них, сообщай пользвателю.

## Комментарии

Для Makefile
используем следующие виды комметариев:
```
### Название группы команд, например внутри контейнера, БД и т.п
## Название конкретной одиночной команды

Например:

### Проверки качества
## Запустить тесты, покрытие, OpenAPI validation и pre-commit
check_host: coverage_host schema_validate_host pre-commit_host

## Запустить pre-commit хуки
pre-commit_host:
	uv run pre-commit run --all-files
```

В Python файлах комментарии
```
# 1. Заголовок первого уровня ----

## 1.1. Заголовок второго уровня ----

```

Во всех файлах можно выделять крупные блоки для удобного скрытия кода
```
Большие блоки

# ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ #
#                            УПРАВЛЕНИЕ КОНТЕЙНЕРОМ                            #
# ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~ #

Тут прочие команды

Или код


# ---------------------------------------------------------------------------- #

```


## Итоговый отчёт

После завершения задачи обязательно предоставь:

### Что было сделано

Краткое описание внесённых изменений.

### Изменённые файлы

Список изменённых файлов.

### Выполненные проверки

Перечень запущенных команд и их результат.

### Непройденные или недоступные проверки

Укажи:

* какие проверки не удалось выполнить;
* по какой причине;
* требуется ли пересборка контейнеров или дополнительная настройка окружения.

### Предложение для коммита

Всегда предлагай готовое название коммита в формате Conventional Commits.

Примеры:

```text
fix: исправить обработку пустого профиля пользователя

feat: добавить фильтрацию заказов по статусу

refactor: упростить логику формирования отчёта

test: добавить регрессионный тест для авторизации

chore: обновить конфигурацию pre-commit

build: обновить настройки docker-compose
```


Коммиты пиши по-русским, кроме fix, feat, refactor и т.п, также в скобках можешь указывать области действия (views-ProductListView)
